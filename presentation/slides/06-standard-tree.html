Limitation

The Standard Semantic Tree Problem

⚙️ How It Works

Single SPARQL query retrieves the entire relationship graph at initialization — synchronous, monolithic loading

Works for:

✓ Small datasets
✓ Few hundred nodes

Fails for:

✗ 6M triples
✗ 18 levels deep
✗ 1000s of entities

⚠️ Critical Issues at Scale

Prohibitive initial loading times

Memory consumption exceeds thresholds

System responsiveness deteriorates

Navigation is strictly vertical — no horizontal semantic exploration

Key insight: We need a fundamentally different architecture — not just optimization, but a paradigm shift in how data is loaded and navigated
Within ResearchSpace, the standard Semantic Tree module is the predefined tool for representing hierarchical relationships among RDF entities. It uses a synchronous, monolithic approach: a single initial query retrieves the ENTIRE relationship graph at once. For small datasets this works fine. But for the Evangelisti archive, with 18 hierarchical levels and thousands of entities across 6 million triples, this strategy proved computationally untenable. Initial loading times became prohibitive, memory consumption exceeded reasonable thresholds, and responsiveness deteriorated. Moreover, navigation is strictly vertical—you can only go up and down the hierarchy, but you cannot explore the horizontal semantic connections that make archival materials rich and interesting.
